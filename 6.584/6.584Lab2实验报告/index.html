    6.584Lab2实验报告

    实验目标

    Build a key/value server for a single machine that ensures that each Put operation is executed at-most-once despite network failures and that the operations are linearizable.

    实验难点

    1. 在实现CLERK的GET/PUT函数的时候，到底什么时候需要重发rpc，比如说，call本身返回的ok message不为空时，我们应该像lab1里面一样直接报错呢，还是重新发一遍rpc，这个的话根据reliable network和unreliable network的说法，就是要在网络不可靠的时候重发的，所以应该问题不大

    2. 现在的问题是为什么无法通过test:"many clients racing to put values to the same key (reliable network)..."，在服务器中的in-memory map 已有读写锁的情况下，为什么会出现这种情况？？？woc，使用读写锁，然后用Rlock锁住map的方法竟然是错误的，必须要使用lock锁住才行，为什么呢，艹，这样是不行的，只是我改完的那一次刚好对了，红温了。一开始，我是在每个读写操作的前后获得锁释放锁，改之后用了defer就可以了。懂了，我对临界区的判断出问题了，不只是只有要进行读写操作的才叫关键代码，我们还要关注逻辑上的连贯性，比如下面这段代码:

    func (kv *KVServer) Put(args *rpc.PutArgs, reply *rpc.PutReply) {
key := args.Key

kv.mu.Lock()
defer kv.mu.Unlock()
value, ok := kv.cache[key] //area 1
if ok {
    if value.ver == args.Version {
        new_value := Value{val: args.Value, ver: args.Version + 1}
        // kv.mu.Lock()
        kv.cache[key] = new_value //area 2
        // kv.mu.Unlock()
        reply.Err = rpc.OK
    } else {
        reply.Err = rpc.ErrVersion
    }
} else {
    if args.Version == 0 {
        new_value := Value{val: args.Value, ver: args.Version + 1}
        // kv.mu.Lock()
        kv.cache[key] = new_value //area 3
        // kv.mu.Unlock()
        reply.Err = rpc.OK
    } else {
        reply.Err = rpc.ErrNoKey
    }
}
}

    错误的做法：只在我标注出来的这三个地方进行并行保护，但是事实上，这段代码具有逻辑上的连贯性，在这几个代码区域中间没有持有锁，如果发生了其它线程对公共区域的修改，就会导致意料之外的情况发生

    1. 实现lock service的过程中，我把本地锁和分布式锁没有区分清楚。要知道对于每一个client来说，都会在本地创建一个自己的锁，所以必须要与server沟通，通过get和put函数实现一个“锁”

    2. 为什么无法在不可靠的网络环境下实现锁？根据打印出来的结果来看，似乎是Get函数一直在试图获取RPC，但是一直失败。通过日志终于发现原来是acquire函数一直在通过GET获取锁，但是获取到的锁的状态都是已曾被自己上过锁，我没有正确地处理这种情况。
