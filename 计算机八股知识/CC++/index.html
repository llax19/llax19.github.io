    c中的用法

    首先是在c中的用法，分为三种情况：修饰局部变量、修饰全局变量、修饰函数。

    • 静态局部变量，会使得原本存储在栈区的变量变成存储在数据段，这样一来，变量的作用域大小没有变，生命周期变得和全局变量一样。并且该变量只会被初始化一次，若未被初始化会默认初始化为0
    • 静态全局变量和函数会使得外部文件无法引用该变量或函数，即改变了全局变量和函数的外部链接属性

    c++中的用法

    • 静态成员变量，必须在类外定义与初始化，在定义时不用加static，要用类名::变量名来定义。静态成员变量为所有类对象所共享，不属于某个具体的实例。
    • 静态成员函数没有隐藏的this指针，不能访问任何非静态成员，只能访问静态成员变量。而非静态成员函数可以调用类的静态成员函数

    const关键字的用法

    C/C++ 基础中的基础： const 修饰符用法总结！ - 知乎

    const变量

    const变量必须被初始化，const结构体变量表示结构体内的任何数据域都不可被改变，且需要另一个结构体变量进行初始化

    const类对象

    const类对象与const变量并无什么区别。要注意的是这里“初始化”和“改变”的定义有些不同。初始化指的就是能成功调用构造函数，而改变指的是改变成员变量以及调用非常量成员函数。

    指向const变量的指针和const指针

    这两个容易让人混淆，因此可以这么理解：从右向左、依次结合、const就近结合

    比如const int * const p可以这么解读：

    1. p是一个变量
    2. const p表示p是一个常量，也就是说p本身的值无法被改变
    3. * const p 表示p是一个指针
    4. int * const p 表示p指向的是一个int
    5. const int * const p表示p指向的是一个常量的int

    当使用指针访问一个变量时，编译器是通过指针类型（const int *）来判断这个变量到底是否只读的。因此使用以下代码，我们可以“改变”一个const变量。

    const int i = 0;
int * q;
q = const_cast<int*>(&i);
*q = 10;
cout<<*q<<" "<<i;

    以上代码是可以通过编译并正常运行的，但是输出结果却不一定。在编译器版本比较低的时候确实能看到输出了两个相同的值，但是编译器版本高了之后会发现 i 的值没有发生变化。这是由于编译器的优化导致实际上当你在访问i的时候编译器会直接使用值的替换，所以在程序的实际运行中不会产生一次变量访问。所以使用const全局变量来代替宏定义具有相同的执行效率并且还能让编译器进行类型检查。

    const成员变量

    const成员变量必须被初始化（在类的构造函数的初始化列表中）。

    类对象的实例化过程可以理解为包含以下步骤：首先分配整个类对象的内存空间，然后分配各个成员变量的内存空间，并根据构造函数的初始化列表进行初始化。最后再执行构造函数的代码。由于const变量在定义时就要进行初始化，所以只能在初始化列表中初始化，然后就不能更改了。

    const返回值

    函数返回const返回值一般用于返回一个const引用，如果只是返回引用那么其实可以直接更改这个引用所指向的值，但是如果是常量引用那就改不了了。

    const成员函数

    const成员函数是指此函数不能修改任何成员变量。传给const函数的this指针是指向const对象的const指针，所以const函数不能修改任何成员变量，除非成员变量被mutable修饰符修饰。

    这里应该注意，由于const成员函数的this指针指向的是const对象，那么如果const成员函数想要调用非const成员函数，就要把自己的this指针传给要调用的函数，但是const变量是不能隐式转换为普通变量的，因此就会出现错误，调用失败。同理，“const对象只能调用const成员函数”也是这个道理。

    .h文件和.hpp文件

    一般来说h文件都只存放声明以及inline函数的定义，而hpp文件更像h文件和cpp文件的混合，可以大幅度减少调用项目中的cpp文件数量和编译次数。同时，引入hpp的一大原因就是模板

    关于模板

    模板更像一种宏，只有在被某个编译模块调用的时候才会在该编译模块初始化出一个示例，模板本身是不会产生汇编代码的，是模板生成的具体化实例才会产生指令

    函数模板的类型检查

    带模板的函数会经历两阶段的检查：

    • 第一阶段检查与类型参数无关的代码
    • 第二阶段发生在模板函数实例化之后，此时检查与类型参数相关的代码

    关于虚函数

    对于每一个实现了虚函数的类来说，都有一个虚表负责存放对应虚函数的函数指针。当我们实例化这个类，在执行这个类的构造函数时，会初始化一个指向这个虚表的虚指针。 虚函数一般会在类的继承中与一般的函数产生区别。例子如下

    #include<iostream>  
using namespace std;  

class A  
{  
public:  
    void foo()  
    {  
        printf("1\n");  
    }  
    virtual void fun()  
    {  
        printf("2\n");  
    }  
};  
class B : public A  
{  
public:  
    void foo()  //隐藏：派生类的函数屏蔽了与其同名的基类函数
    {  
        printf("3\n");  
    }  
    void fun()  //多态、覆盖
    {  
        printf("4\n");  
    }  
};  

class C : class B
{
public:
    void foo() {
        printf("5\n");
    }
    void fun() { //此时还是可以实现虚函数的多态覆盖
        printf("6\n");
    }
}
int main(void)  
{  
    A a;  
    B b;  
    A *p = &a;  
    p->foo();  //输出1
    p->fun();  //输出2
    p = &b;  
    p->foo();  //取决于指针类型，输出1
    p->fun();  //取决于对象类型，输出4，体现了多态
    return 0;  
}

    第一个是子类创建对象时会继承基类的private变量，但是这对于子类来说时不可见的，只是占用了内存空间（这里的原因是你可以把指向子类的指针转化成父类类型的指针，这样就要求你可以访问原先private的成员变量！），这里还有一点要注意，一个空类对象（也就是没有成员变量和虚函数的类），大小是1，为什么布什0呢，因为实例化要求每个实例在内存中都有一个独一无二的地址，为了达到这个目的，编译器往往会给空类隐含地添加一个字节；

    关于访问私有成员

    事实上，只要你在类内或者已声明的友类内，就可以访问私有变量，不管这个私有变量是不是属于这个实例对象（也就是说也可以是其它实例的），而如果你在类外，那就是不能访问，由编译器来维护这样的规则

    关于minisql项目

    • 在内存管理方面：首先，minisql基于磁盘中的文件构建了一整个数据库，所有页的信息都以page的形式存在文件里面，这个文件里面的page分为元数据页和真正存储数据的数据页，这个文件暴露给上层的接口都只接受逻辑页号，也就是将真正的数据页视为连续的，这就需要管理这个文件的类在内部处理时将其提前转为物理页号。而在内存中，数据库维持一个buffer pool，这里面最关键的数据结构是一个page数组，page类相当于一个容器，可以放进去不同的页的数据，page在数组中的下标相当于所谓的“frame id”，上层所使用的相当于“logical id”，我们同样维持一个页表对其的映射关系进行记录。值得一提的是，这里的logical id就是磁盘文件所呈现的logical id。
    • 在记录管理方面，一个表最多只支持32个属性，这个是为了实现上的便利，因为在定义一条记录时，里面的一项可能为空，这个时候就需要使用一个bitmap去记录哪些项可能为空，还有一方面就是使用堆表管理记录的时候，我们为了加快删除速度，都是要先将其标记为删除的，若不怎么让它支持更多属性？

    计网

    TCP三次握手

    过程：

    • 服务器创建传输控制块，被动打开连接，进入LISTEN状态
    • 客户主动打开连接，发送SYN=1，seq=x的报文段，进入SYN-SENT状态，SYN为1的报文段不能携带数据
    • 服务端收到后发送SYN=1，ACK=1，seq=y，ack=x+1的报文段，进入SYN-RCVD状态，与上面同理
    • 客户端发送ACK=1,，seq=x+1，ack=y+1的报文段，进入ESTABLISH状态，这个可以携带数据，但如果不携带数据，就不消耗一个序号，数据报文段的seq就仍是x+1
    • 服务端收到后进入ESTABLISH状态

    TCP和UDP的区别

    • TCP是可靠的、面向连接的流协议。流是指不间断的数据结构，TCP可以保证应用层传下来的数据发送的顺序，但是会像没有间隔一样地发送给接收端。TCP把应用层交付的数据视为一连串的无结构的字节流，TCP和应用层之间有缓冲区，当应用层一次性交付的数据太多了的时候，TCP可以将它划分后再发送。TCP协议实现了许多数据传输时的控制功能，比如丢包重传、对次序乱掉的包进行顺序控制。

    • UDP是不可靠的、面向报文的数据报协议。面向报文是指无论应用层一次性交给UDP多长的报文，UDP都会照样一次性发送。所以必须由应用层选择合适大小的报文。

    • TCP应用：HTTP、SMTP、FTP、TELNET（用于远程接入）

    • UDP应用：DNS、SNMP（网络管理）

    OSI七层模型

    应用层：为应用程序或用户请求提供各种请求服务。OSI参考模型最高层，也是最靠近用户的一层，为计算机用户、各种应用程序以及网络提供接口，也为用户直接提供各种网络服务。

    表示层：数据编码、格式转换、数据加密。提供各种用于应用层数据的编码和转换功能,确保一个系统的应用层发送的数据能被另一个系统的应用层识别。如果必要，该层可提供一种标准表示形式，用于将计算机内部的多种数据格式转换成通信中采用的标准表示形式。数据压缩和加密也是表示层可提供的转换功能之一。

    会话层：创建、管理和维护会话。接收来自传输层的数据，负责建立、管理和终止表示层实体之间的通信会话，支持它们之间的数据交换。该层的通信由不同设备中的应用程序之间的服务请求和响应组成。

    传输层：数据通信。建立主机端到端的链接，为会话层和网络层提供端到端可靠的和透明的数据传输服务，确保数据能完整的传输到网络层。

    网络层：IP选址及路由选择。通过路由选择算法，为报文或通信子网选择最适当的路径。控制数据链路层与传输层之间的信息转发，建立、维持和终止网络的连接。数据链路层的数据在这一层被转换为数据包，然后通过路径选择、分段组合、顺序、进/出路由等控制，将信息从一个网络设备传送到另一个网络设备。

    数据链路层：提供介质访问和链路管理。接收来自物理层的位流形式的数据，封装成帧，传送到网络层；将网络层的数据帧，拆装为位流形式的数据转发到物理层；负责建立和管理节点间的链路，通过各种控制协议，将有差错的物理信道变为无差错的、能可靠传输数据帧的数据链路。

    TCP的流量控制和拥塞控制

    • TCP的流量控制：流量控制是为了解决接收方和发送方处理数据的速度的差异导致的数据丢失问题，当发送方发送数据的速度太快时，接收方的缓冲区就有可能发生数据溢出导致数据丢失。 解决方法是使用滑动窗口，在TCP的首部中维持一个window字段，接收方用其指明自己的缓冲区还有多少空余空间。当接收方声明自己的缓冲区余量为0后，发送方即不再向接收方发送数据直到再一次接收到接收方具有余量的通知。若一段时间后发送方还没有收到接收方的通知，则会发送一个空数据包用于探测最新的窗口数据。
    • 拥塞的定义：在某段时间内，外部向网络注入的数据量远超网络内资源可以处理的限度，导致网络性能急剧下降，叫做拥塞。 TCP的发送方维护了一个cwnd窗口，该窗口反映了发送方对于网络拥塞情况的认知（实际上的发送窗口大小取cwnd与接收方通知的窗口值中的小值）。ssthresh是指慢开始门限，是为了防止cwnd增长过快所设置的门限值。一般通过四种算法实现拥塞控制管理：
    • 慢开始：当swnd<ssthresh时，从窗口大小为1的窗口开始，然后翻倍增长
    • 拥塞避免：当cwnd>ssthresh时，每次只增长1
    • 快重传：接收方即使收到失序的报文段也要立即发出对已接收报文段的确认。发送方在收到接收方的连续三个重复确认后立即开始重传
    • 快恢复：快重传之后cwnd不从1开始，从ssthresh/2开始

    TCP三握手

    过程：

    • 服务器创建传输控制块，被动打开连接，进入LISTEN状态
    • 客户主动打开连接，发送SYN=1，seq=x的报文段，进入SYN-SENT状态，SYN为1的报文段不能携带数据，但小号一个序号
    • 服务端收到后发送SYN=1，ACK=1，seq=y，ack=x+1的报文段，进入SYN-RCVD状态，与上面同理，也要消耗一个序号
    • 客户端发送ACK=1,，seq=x+1，ack=y+1的报文段，进入ESTABLISH状态，这个可以携带数据，但如果不携带数据，就不会消耗一个序号，数据报文段的seq就仍是x+1
    • 服务端收到后进入ESTABLISH状态

    这里引用阿里云上一篇文章里的内容解释一下为啥只有三次握手才能确认双方的接受与发送能力是否正常，而两次却不可以：

    • 第一次握手：客户端发送网络包，服务端收到了。这样服务端就能得出结论：客户端的发送能力、服务端的接收能力是正常的。
    • 第二次握手：服务端发包，客户端收到了。这样客户端就能得出结论：服务端的接收、发送能力，客户端的接收、发送能力是正常的。不过此时服务器并不能确认客户端的接收能力是否正常。
    • 第三次握手：客户端发包，服务端收到了。这样服务端就能得出结论：客户端的接收、发送能力正常，服务器自己的发送、接收能力也正常。

    我们也可以从另一个角度看，就是两次握手的话会发生什么： 两次握手意味着服务器发出ack报文之后就进入ESTABLISH状态。此时如果第一次客户端的连接请求被延误了导致重发，重发后两者成功建立连接，一段时间后成功断开。若此时第一次被延误的请求到达了服务端，那么服务端会发出ACK报文后进入连接，但是此时客户端关闭，无法进入连接，这样服务端的资源就被空置了。

    TCP四挥手

    过程：

    • 客户端主动关闭连接，发送FIN报文，进入FIN-WAIT1状态
    • 服务器接收并发送ACK=1的报文，此时应用程序开始发送最后的数据，进入CLOSE-WAIT状态
    • 客户端接收并等待，进入FIN-WAIT2状态
    • 服务器发送完数据后发送FIN=1,ACK=1的报文，进入LAST-ACK状态
    • 客户端接收并发送ACK=1的报文，进入TIME-WAIT状态，并维持2MSL的时间后关闭
    • 原因：如果该ACK报文丢失，服务器会重发FIN报文，此时客户端需要重发该报文，若不等待直接关闭可能导致服务器资源空置
    • 服务器接收到后关闭

    问：什么是线性一致性

    答：线性一致性是在服务器面对并发的客户端请求这样的场景下对于正确性行为的定义。粗略地来讲，它要求服务器处理的结果应该和按照请求到来的时间串行处理的结果保持一致。

    线性一致性应该是基于“历史”实现的，这里的“历史”记录了客户端开始这个操作的时间和客户端看见这个操作结束的时间，线性一致性就是定义了什么样的历史记录是合理的，以及如果一个服务器生成的历史都是线性一致的，那么我们称这个服务器是线性一致的。

    对于客户端开始操作和客户端判定操作完成这两个事件，历史记录会分别进行记录。从而使历史记录明确显示客户端之间的并发情况和网络延迟。通常情况下，这两个开始与结束事件分别对应客户端与服务器之间交换的请求消息和响应消息。

    线性化点：对于一条操作来说，线性化点位于它的开始点和结束点之间。如果存在一种线性化点的分布可以使得这个历史按照线性点的顺序执行每条操作后得到的对应值和原历史的一样，那么我们说这个历史是线性一致的。

    线性一致性的一个重要结果是：服务可以自由决定并发（时间上重叠）操作的执行顺序。具体来说，若客户端 C1 和 C2 的操作是并发的，即使 C1 的操作先于 C2 启动，服务器仍可先执行 C2 的操作。另一方面，如果 C1 的操作在 C2 启动前就已完成，线性一致性要求服务必须表现出先执行 C1 操作的行为（即 C2 的操作必须能观察到 C1 操作的所有影响，若有的话）。

    问：线性一致性 checker 是如何工作的

    一个简单的线性一致性检查器会尝试所有可能的顺序（或线性化点的选择），以验证是否存在符合线性一致性定义规则的有效顺序。由于这种方法在大型历史记录上速度太慢，聪明的检查器会采取以下优化措施：

    1. 跳过明显不可能的顺序（例如拟议的线性化点早于操作的开始时间）
    2. 在可能的情况下将历史分解为可独立验证的子历史
    3. 使用启发式方法优先尝试更可能的顺序
